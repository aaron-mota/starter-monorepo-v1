Project Overview

A production-ready full-stack SaaS starter

A Turborepo monorepo with a Next.js 16 web app, an Expo React Native mobile app, and shared TypeScript packages — auth, billing, dashboard, API layer, and mobile, all wired up with opinionated conventions and quality tooling out of the box.

Next.js 16 React 19 React Native (Expo) TypeScript 6 tRPC MongoDB Clerk Stripe Zod 4 Turborepo pnpm

What this is

An opinionated starter for SaaS-style projects: marketing site, authenticated dashboard, subscription billing, and a companion mobile app — all sharing a single source of truth for schemas and types. Designed to be cloned, renamed, and shipped without ripping out half-built systems.

Three-tier schema pattern

Every database entity lives in packages/shared/schemas/database/<entity-name>/ with a strict three-layer split: client/ (what React components use), api/ (what crosses the network as JSON), and server/ (what MongoDB stores). Each layer has its own Zod schemas, inferred TypeScript types, and adapter functions that bridge to the next. There's no ORM in the way — the schemas in server/schema.ts are the model, and queries hit the official MongoDB driver directly. 

packages/shared/schemas/database/user/
├── _config.ts              # Root TYPE + barrel exports (client + api only)
├── client/                 # FE — what React components consume
│   ├── _config.ts          # SCHEMA, SCHEMA_CREATE, SCHEMA_UPDATE
│   ├── schema.ts           # schema, schemaCreate, schemaUpdate (Zod)
│   ├── types.ts            # TDoc, TDocCreate, TDocUpdate
│   └── constants.ts        # DEFAULT_VALUES, KEYS_*
├── api/                    # wire format — what JSON over the network looks like
│   ├── _config.ts
│   ├── schema.ts           # schemaApi, schemaApiCreate, schemaApiUpdate
│   ├── types.ts            # TDocApi, TDocApiCreate, TDocApiUpdate
│   └── adapterFns.ts       # adapterFnApiToFE, adapterFnFEToApiCreate, ...
└── server/                 # DB — what gets stored in MongoDB (server-only)
    ├── _config.ts          # import "server-only"; — guards the boundary
    ├── schema.ts           # schemaDb, schemaDbCreate, schemaDbUpdate
    ├── types.ts            # TDocDb, TDocDbCreate, TDocDbUpdate
    └── adapterFns.ts       # adapterFnDbToFE, adapterFnFEToDbCreate, ...

Why three layers?

The boundaries between FE / wire / DB don't carry the same types. ObjectId can't serialize to JSON; Date objects come back from the network as ISO strings; the FE wants id: string and real Date instances. Each adapter does exactly one of those translations, in one place, at one boundary:

Layer Shape of id Shape of dates Used by
client id: string Date React components (web + mobile), forms, hooks
api _id: string ISO strings JSON in flight (REST endpoints, webhook payloads)
server _id: ObjectId Date MongoDB driver in @app/db

A typical read flows DB → wire → FE via adapterFnDbToFE on the server, then JSON serialization across the network produces the api shape, then adapterFnApiToFE rehydrates it on the client (string _id → id, ISO strings → Date). A write reverses it.

server/ is firewalled. Every server/ file starts with import "server-only". Entity barrels (_config.ts) only re-export from client/ and api/ — never server/ — so a client-side import { TDoc } from '@app/shared' can never accidentally pull in BSON code.
Note: tRPC hides the wire layer for you. tRPC is configured with the superjson transformer (in packages/trpc/src/trpc-server.ts), which serializes & deserializes Date (plus BigInt, Map, Set, etc.) transparently across the network. For an end-to-end tRPC call, the client receives real Date instances back — no ISO-string detour, no manual rehydration. As long as id is a pre-stringified ObjectId, the client/ + server/ split alone is sufficient.

The api/ layer becomes load-bearing the moment you leave that bubble:
  • exposing public REST endpoints under apps/web/src/app/api/
  • receiving webhooks (Stripe, Resend, Clerk) where the payload is plain JSON
  • integrating with external APIs whose responses follow their own wire format
  • any code path where the network carries plain JSON instead of superjson-encoded JSON
In those cases Date ↔ ISO string and ObjectId ↔ string have to be handled explicitly. The api/ folder makes that boundary type-safe and one-place-to-fix instead of letting it leak into every fetch call.
Adapters are also where you reshape data, not just convert types. Each adapter is a normal function with full freedom to transform what passes through. That makes the DB shape and the FE shape independent on purpose — you can decide what the FE sees without dragging storage decisions along with it. Things you can do at this boundary:
  • Rename fields — canonical camelCase on the FE even if the DB column is created_by_id; or rename a clumsy DB-side name to something more readable
  • Exclude fields — drop sensitive data (password hashes, internal IDs), denormalized cache columns, or anything the UI never needs
  • Add computed fields — surface a derived fullName built from firstName + lastName, an isOverdue flag from a date, a relative age, etc.
  • Combine flat columns into structured objects — turn address1, address2, city, state, zip into a nested address object on the FE
  • Coerce formats — priceInCents: 1299 in DB → price: 12.99 on the FE; ISO date strings → Date instances; etc.
The user-model adapter (packages/shared/schemas/database/user/server/adapterFns.ts) is a small real example: it strips _id from the DB shape and produces id: string on the FE, so the FE never has to know about MongoDB's ObjectId at all.
FYI: the same boundaries make swapping the database tractable. Because the FE only ever sees client/ types (TDoc with id: string and real Date instances), it doesn't know or care what's storing the data underneath. There's no ORM in the way either — the Zod schemas in server/schema.ts are the model. Migrating from MongoDB to Postgres (Drizzle, Prisma, etc.) is mostly a server/-folder job per entity:
  • update server/schema.ts & server/types.ts to describe the new row shape (e.g. id: serial instead of an ObjectId-shaped string)
  • rewrite server/adapterFns.ts to map the new DB shape back to TDoc
client/ and api/ stay untouched, components stay untouched, forms stay untouched.

Caveats, to be honest: the cross-cutting pieces would still need work — the connection layer in packages/db/src/, the tRPC context that resolves the current user via Mongo, the generic base-router helpers that take raw Mongo queries (getManyMongo, getSingleMongo) would need an equivalent for the new engine, and Mongo-only deps would come out of package.json. But the per-entity migration is contained, and the FE doesn't flinch.

Workspace packages

Each package has a single, narrow responsibility. Inter-package dependencies use workspace:* via pnpm so internal versions are always in lockstep.

@app/shared

Schemas, types, constants

Single source of truth: Zod schemas for every entity, inferred TypeScript types, and shared constants like PLAN_LIMITS and category metadata. Imported by every other package and both apps.

no runtime deps on db / clerk / etc.

@app/db

MongoDB layer

Native MongoDB driver (no Mongoose). Typed collection wrappers, index definitions, and a db:indexes setup script. Server-only by construction — every file starts with import "server-only".

depends on: @app/shared

@app/trpc

tRPC routers

Routers and a reusable createBaseCrudRouter() that wires standard get, getMany, create, update, delete procedures for any entity from its schemas.

depends on: @app/db

@app/api

Client hooks

React Query hooks layered on top of tRPC procedures, plus a useBasicCrudProcedure helper that mirrors the server-side base router. Used by both web and mobile.

depends on: @app/trpc

@app/ui

Shared UI primitives

Intentionally minimal — the web app uses shadcn/ui directly (owned, not imported), and mobile uses rn-primitives. This package is the seam where truly cross-platform components would live if needed.

scaffolded

Code conventions

Enforced by ESLint, Prettier, and code review. CLAUDE.md captures these for AI assistants.

a-z
kebab-case file names — only

my-component.tsx, never MyComponent.tsx. use-my-hook.ts, never useMyHook.ts. Folders too.

ex
Named exports

export function Foo() — never export default. Exception: Next.js page components, which the framework requires to be default exports.

ui
shadcn/ui components, not raw HTML

<Button> over <button>, <Input> over <input>, <Skeleton> for loading, <Alert> for errors.

∀
Every component handles four states

Loading (<Skeleton>), Error (<Alert variant="destructive">), Empty (with meaningful copy), and Success. No "happy path only" components.

↻
Pull-to-refresh on every mobile data screen

Use the useRefresh hook — {' on ScrollView, or built-in props on FlatList. Calls utils.invalidate() to refetch all queries.

3✕
Three-tier schema separation

Every database entity has client/, api/, and server/ folders with explicit adapter functions between layers. Zod is the source of truth — types are inferred, never written by hand.

⊥
No Mongoose

Native mongodb driver only. Schemas live in @app/shared, not in model classes that try to be both validators and ORMs.

🔒
"server-only" guard on every server file

Every file under server/ opens with import "server-only". Entity barrels never re-export from server/, so client bundles can't drag in ObjectId / BSON code.

Conventional commits

Commit messages use feat:, fix:, chore:, refactor:, test:, docs:, style: prefixes. Pre-commit hook runs lint-staged (ESLint + Prettier on changed files) and pnpm type-check; pre-push hook runs a full type-check across the workspace.

Customizing for your project

  1. Rename @app/* packages to @yourproject/* across all package.json files and imports.
  2. Update branding (app name, icons, colors in globals.css).
  3. Replace the example Item entity with your domain entities, following the three-tier schema pattern.
  4. Add your pages, components, and tRPC routers.
  5. Configure Stripe products and prices for your pricing tiers.

Quality tooling

turbo.json

Turborepo

Modern tasks syntax (not legacy pipeline). Cached build, lint, type-check, test:run. dev marked persistent + uncached. Vercel deploy uses turbo-ignore to skip rebuilds when only mobile changed.

tsconfig.base.json

TypeScript 6.0 (strict)

Strict mode on. Modern JSX transform (react-jsx). Each package extends the base config and overrides only what it needs (e.g., mobile uses Expo's tsconfig).

eslint.config.mjs

ESLint 9 (flat config)

Flat config, not legacy .eslintrc. Rules from typescript-eslint and eslint-plugin-react. shadcn/ui components in components/ui/ are excluded — they're owned but not authored.

.prettierrc

Prettier

Tailwind-aware (prettier-plugin-tailwindcss sorts class names) and import-sort plugin keeps imports tidy. Runs via lint-staged on commit and pnpm format on demand.

vitest.config.ts

Vitest + Testing Library

ESM-native, jsdom environment, mocks for Clerk and Next.js navigation in the test setup. Tests co-located in __tests__/ directories. Sample tests cover helpers (slug generation), shared components (PageHeader, EmptyState), and dashboard cards.

.husky/

Git hooks

pre-commit: lint-staged (ESLint + Prettier on changed files) then pnpm type-check.
pre-push: full pnpm build across the workspace. Both hooks must pass before code lands.

vercel.json

Vercel deploy

Monorepo-aware: buildCommand uses turbo build --filter=@app/web... so only the web app and its deps build. turbo-ignore skips the build entirely for changes that don't affect web.

pnpm-workspace.yaml

pnpm workspaces

apps/* and packages/* globbed. node-linker=hoisted in .npmrc for compatibility with tooling that doesn't grok pnpm's symlinked layout (notably parts of the React Native + Expo stack).

Production build flow

  1. git commit → pre-commit hook runs lint-staged + type-check
  2. git push → pre-push hook runs pnpm build (full Next.js production build)
  3. Vercel webhook fires → turbo-ignore checks if web changed
  4. If yes, turbo build --filter=@app/web... runs with cached deps
  5. Edge functions, middleware, and static assets ship
Cache hygiene: Turborepo's content-addressed cache means a no-op change to @app/web can replay logs from a hit in milliseconds — but a change to @app/shared correctly invalidates everything downstream.
